# Лабораторна робота №4
## Організація параллельної обробки запитів з багатопроцесовим сервером

### Завдання
Модифікувати сервер з Л.р. №1 для паралельного обслуговування клієнтів за допомогою **процесів**.

### Варіант: 1 (номер_по_журналу mod 17 = 1)

| Параметр | Значення |
|----------|----------|
| Протокол | TCP |
| Порождення | Процеси по запиту |
| Механізм захисту | Відсутній |

### Архітектура сервера

```
                    ┌─────────────────┐
                    │  Main Process   │
                    │   (Accept Loop) │
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
              ▼              ▼              ▼
        ┌──────────┐  ┌──────────┐  ┌──────────┐
        │ Process 1│  │ Process 2│  │ Process 3│
        │ Client A │  │ Client B │  │ Client C │
        └──────────┘  └──────────┘  └──────────┘
```

### Реалізовані механізми

#### 1. Динамічний пул процесів
- **Nmin = 2** - мінімальна кількість процесів
- **Nmax = 10** - максимальна кількість процесів
- Процеси створюються при підключенні клієнта
- Зайві процеси завершуються при простої

#### 2. Спільна статистика (SharedStats)
```python
class SharedStats:
    active_clients      # Активні клієнти
    total_connections   # Всього підключень
    bytes_transferred   # Передано байт
    processes_created   # Створено процесів
```

#### 3. Таймаути
- **PROCESS_TIMEOUT = 30с** - час життя простоючого процесу
- **CLIENT_TIMEOUT = 300с** - таймаут неактивності клієнта

### Порівняння з Л.р. №3 (Мультиплексування)

| Характеристика | select() (Л.р. №3) | Процеси (Л.р. №4) |
|---------------|-------------------|-------------------|
| Потoki | 1 | N (на клієнта) |
| Пам'ять | Мінімальна | N × 2-4 MB |
| Перемикання контексту | Ні | Так |
| Ізоляція | Ні | Так |
| Стабільність | Crash = всі | Crash = один |
| Масштабованість | До 1024 FD | Обмежено RAM |

### Переваги багатопроцесовості

1. **Ізоляція**
   - Crash одного процесу не впливає на інші
   - Кожен процес має власну пам'ять

2. **Простота**
   - Блокуючий I/O
   - Не потрібно обробляти часткові читання

3. **Використання багатоядерності**
   - Процеси розподіляються по ядрах
   - Паралельне виконання

### Недоліки

1. **Витрати пам'яті**
   - Кожен процес: 2-4 MB
   - 100 клієнтів = 200-400 MB

2. **Перемикання контексту**
   - Дорожче ніж перемикання потоків
   - Зниження продуктивності при великій кількості

3. **Міжпроцесна комунікація (IPC)**
   - Повільніша ніж спільна пам'ять
   - Потрібна синхронізація

### Питання для захисту

1. **Чому процеси а не потоки?**
   - Ізоляція помилок
   - Використання багатоядерності (GIL в Python)

2. **Як працює SharedStats?**
   - multiprocessing.Value для спільних змінних
   - Lock для синхронізації доступу

3. **Навіщо Nmin і Nmax?**
   - Nmin: швидка відповідь при різкому навантаженні
   - Nmax: захист від вичерпання пам'яті

4. **Як завершуються процеси?**
   - При відключенні клієнта
   - При простої (timeout)
   - При scale down

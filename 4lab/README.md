# Лабораторная работа №4
## Организация параллельной обработки запросов с многопроцессным сервером

### Задание
Модифицировать сервер из Л.р. №1 для параллельного обслуживания клиентов с помощью **процессов**.

### Вариант: 1 (номер_по_журналу mod 17 = 1)

| Параметр | Значение |
|----------|----------|
| Протокол | TCP |
| Порождение | Процессы по запросу |
| Механизм защиты | Отсутствует |

### Архитектура сервера

```
                    ┌─────────────────┐
                    │  Main Process   │
                    │   (Accept Loop) │
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
              ▼              ▼              ▼
        ┌──────────┐  ┌──────────┐  ┌──────────┐
        │ Process 1│  │ Process 2│  │ Process 3│
        │ Client A │  │ Client B │  │ Client C │
        └──────────┘  └──────────┘  └──────────┘
```

### Реализованные механизмы

#### 1. Динамический пул процессов
- **Nmin = 2** - минимальное количество процессов
- **Nmax = 10** - максимальное количество процессов
- Процессы создаются при подключении клиента
- Лишние процессы завершаются при простое

#### 2. Общая статистика (SharedStats)
```python
class SharedStats:
    active_clients      # Активные клиенты
    total_connections   # Всего подключений
    bytes_transferred   # Передано байт
    processes_created   # Создано процессов
```

#### 3. Таймауты
- **PROCESS_TIMEOUT = 30с** - время жизни простаивающего процесса
- **CLIENT_TIMEOUT = 300с** - таймаут неактивности клиента

### Сравнение с Л.р. №3 (Мультиплексирование)

| Характеристика | select() (Л.р. №3) | Процессы (Л.р. №4) |
|---------------|-------------------|-------------------|
| Потоки | 1 | N (на клиента) |
| Память | Минимальная | N × 2-4 MB |
| Переключение контекста | Нет | Да |
| Изоляция | Нет | Да |
| Стабильность | Crash = все | Crash = один |
| Масштабируемость | До 1024 FD | Ограничено RAM |

### Преимущества многопроцессности

1. **Изоляция**
   - Crash одного процесса не влияет на другие
   - Каждый процесс имеет собственную память

2. **Простота**
   - Блокирующий I/O
   - Не нужно обрабатывать частичные чтения

3. **Использование многоядерности**
   - Процессы распределяются по ядрам
   - Параллельное выполнение

### Недостатки

1. **Затраты памяти**
   - Каждый процесс: 2-4 MB
   - 100 клиентов = 200-400 MB

2. **Переключение контекста**
   - Дороже чем переключение потоков
   - Снижение производительности при большом количестве

3. **Межпроцессная коммуникация (IPC)**
   - Медленнее чем общая память
   - Нужна синхронизация

### Вопросы для защиты

1. **Почему процессы а не потоки?**
   - Изоляция ошибок
   - Использование многоядерности (GIL в Python)

2. **Как работает SharedStats?**
   - multiprocessing.Value для общих переменных
   - Lock для синхронизации доступа

3. **Зачем Nmin и Nmax?**
   - Nmin: быстрый ответ при резком нагружении
   - Nmax: защита от исчерпания памяти

4. **Как завершаются процессы?**
   - При отключении клиента
   - При простое (timeout)
   - При scale down

### Примечания к вариантам

| Вар | Прот. | Порождение | Механизм защиты |
|-----|-------|------------|-----------------|
| 1 | TCP | Процессы по запросу | - |
| 2 | TCP | Потоки по запросу | - |
| 3 | UDP | Потоки по запросу, один поток = один запрос | - |
| 4 | UDP | Потоки по запросу, один поток = одна сессия | MSG_PEEK |
| 5-11 | TCP/UDP | Пул потоков* | Различные механизмы |
| 12-13 | TCP | Пул процессов* | Блокировка файла |
| 14-17 | TCP/UDP | Процессы/потоки | Мультисервис (порты) |

* -- Поток(процесс)-менеджер следит только за числом свободных потоков (процессов). Пул инициализируется Nmin. При возрастании числа запросов пул динамически расширяется до Nmax. При завершении обработки лишние процессы завершаются после таймаута.

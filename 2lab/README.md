# Лабораторна робота №2
## Клієнт-серверна програма для передачі файлів по UDP

### Завдання
Модифікувати програму з Л.р. №1 для роботи по протоколу UDP з надійною доставкою.

### Реалізовані механізми

#### 1. Підтвердження передачі пакетів (ACK)
- Кожен отриманий пакет підтверджується ACK пакетом
- Формат пакету: тип(1) + seq_num(4) + total_size(4) + checksum(4) + дані

#### 2. Повторна передача при втраті
- Таймаут очікування ACK: 2 секунди
- Максимальна кількість спроб: 5
- При NACK - миттєва ретрансмісія

#### 3. Ковзне вікно
- Розмір вікна: 10 пакетів
- Дозволяє відправляти пакети без очікування ACK на кожен
- Збільшує пропускну здатність

### Оптимальний розмір буфера

**1472 байти** - оптимальний розмір для UDP:
- MTU Ethernet: 1500 байт
- Заголовок IP: 20 байт
- Заголовок UDP: 8 байт
- Корисні дані: 1500 - 20 - 8 = **1472 байти**

### Порівняння продуктивності

| Протокол | Продуктивність |
|----------|---------------|
| TCP (Л.р. №1) | ~100 Kbps |
| UDP (Л.р. №2) | ~150-200 Kbps |

**UDP швидший в 1.5-2 рази** завдяки:
- Відсутності встановлення з'єднання (3-way handshake)
- Меншим накладним витратам на заголовки
- Відсутності вбудованого flow control

### Файли
- `udp_transfer.py` - UDP передача файлів

### Запуск

#### Сервер (приймання файлу):
```bash
python udp_transfer.py server [port]
```

#### Клієнт (відправка файлу):
```bash
python udp_transfer.py client <host> [port]
```

### Тестування втрати пакетів

#### Windows Firewall - DROP пакети:
```powershell
netsh advfirewall firewall add rule name="Drop UDP" dir=in action=block protocol=UDP localport=8080
```

#### Linux iptables - DROP:
```bash
iptables -A INPUT -p udp --dport 8080 -j DROP
```

#### Linux iptables - REJECT:
```bash
iptables -A INPUT -p udp --dport 8080 -j REJECT
```

### Питання 1: Вибір оптимального розміру буфера

**Оптимальний розмір: 1472 байти**

Обґрунтування:
1. **MTU Ethernet** - максимальний розмір кадру: 1500 байт
2. **Заголовки**:
   - IP header: 20 байт (без опцій)
   - UDP header: 8 байт
   - Разом: 28 байт
3. **Фрагментація**:
   - При розмірі > 1472 байт відбувається IP фрагментація
   - Фрагментація збільшує ймовірність втрати пакетів
   - Втрата одного фрагмента = втрата всього пакету

**Обмеження Ethernet**:
- Максимальний розмір кадру (MTU): 1500 байт
- Мінімальний розмір: 64 байти
- При перевищенні MTU - IP фрагментація

### Структура пакету

```
+--------+--------+--------+--------+
|  Type  | Seq Num |Total Size|Checksum|
+--------+--------+--------+--------+
|            Data (max 1472)          |
+-------------------------------------+
```

### Типи пакетів
- `PKT_START (0x01)` - початок передачі
- `PKT_DATA (0x02)` - дані файлу
- `PKT_ACK (0x03)` - підтвердження
- `PKT_NACK (0x04)` - запит ретрансмісії
- `PKT_END (0x05)` - кінець передачі

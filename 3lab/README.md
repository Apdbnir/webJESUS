# Лабораторная работа №3
## Организация параллельной обработки запросов на сервере с помощью мультиплексирования

### Задание
Реализовать в сервере "параллельное" обслуживание нескольких клиентов с помощью мультиплексирования (методы select, pselect, poll).

### Требования
- Сервер работает в рамках **одного потока**
- Последовательное исполнение запросов клиентов
- Отправка/приём порций данных от клиентов
- Подключение новых клиентов без блокировки
- Размер порции: отклик на команды ≤ t = ping × 10
- Обработка команд не прерывает передачу файлов других клиентов

### Реализация

#### Мультиплексирование с select()
- Сервер работает в **одном потоке**
- Одновременная обработка **до 100 клиентов**
- Черёдование между: принятием соединений, чтением данных, отправкой ответов

#### Архитектура сервера

```
┌─────────────────────────────────────────┐
│         MultiplexedServer               │
│  ┌─────────────────────────────────┐    │
│  │         Main Loop               │    │
│  │  ┌──────────────────────────┐   │    │
│  │  │    select(read_fds)      │   │    │
│  │  └──────────┬───────────────┘   │    │
│  │             │                    │    │
│  │    ┌────────┼────────┐          │    │
│  │    ▼        ▼        ▼          │    │
│  │  Accept  Read CMD  Transfer     │    │
│  │    │        │        │          │    │
│  │    └────────┴────────┘          │    │
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘
```

#### Класс Client
Каждый клиент имеет:
- `sock` - сокет соединения
- `buffer` - буфер входных данных
- `state` - состояние (IDLE, RECEIVING_CMD, TRANSFERRING)
- `transfer_data` - данные передачи файла
- `last_activity` - время последней активности

### Команды

| Команда | Описание |
|---------|----------|
| `ECHO <text>` | Возвращает текст клиенту |
| `TIME` | Возвращает время сервера |
| `UPLOAD <file>` | Загрузить файл на сервер |
| `DOWNLOAD <file>` | Скачать файл с сервера |
| `CLOSE/EXIT/QUIT` | Закрыть соединение |

### Производительность

#### Время отклика
- **target**: ping × 10
- **реализация**: RESPONSE_TIMEOUT = 50ms
- Для локальной сети (ping < 1ms): отклик < 10ms

#### Размер порции
- **BUFFER_SIZE = 1024 байта**
- Обеспечивает быстрый отклик на интерактивные команды
- Не блокирует обработку других клиентов

#### Максимальное количество клиентов
- Ограничено `FD_SETSIZE` (обычно 1024 на Linux, 64 на Windows select)
- Реализовано: MAX_CLIENTS = 100

### Преимущества мультиплексирования

1. **Экономия ресурсов**
   - Один поток вместо потока на клиента
   - Меньше переключений контекста
   - Меньше потребление памяти

2. **Масштабируемость**
   - Возможность обслуживать тысячи клиентов
   - Линейное увеличение производительности

3. **Отсутствие race conditions**
   - Нет общих переменных между потоками
   - Проще отладка

### Недостатки

1. **Блокировка на длинных операциях**
   - Решение: неблокирующий I/O + буферизация

2. **Не использует многоядерность**
   - Решение: комбинация с multiprocessing

### select() vs poll() vs epoll()

| Метод | Linux | Windows | Масштабируемость |
|-------|-------|---------|-----------------|
| select() | ✓ | ✓ | До 1024 FD |
| poll() | ✓ | ✗ | До 10000 FD |
| epoll() | ✓ | ✗ | До 1M+ FD |
| IOCP | ✗ | ✓ | До 1M+ FD |

### Вопросы для защиты

1. **Почему select() а не потоки?**
   - Экономия памяти (нет накладных расходов на потоки)
   - Нет race conditions
   - Проще отладка

2. **Какие ограничения у select()?**
   - FD_SETSIZE (обычно 1024)
   - O(n) сложность проверки всех дескрипторов

3. **В чём разница select() vs poll() vs epoll()?**
   - select() - ограничено FD_SETSIZE, O(n)
   - poll() - нет ограничения по FD, O(n)
   - epoll() - O(1) для готовых дескрипторов

4. **Почему малый размер порции (1024 байта)?**
   - Быстрый отклик на интерактивные команды
   - Не блокирует других клиентов
   - ping × 10 = 1ms × 10 = 10ms на команду
